# DEVELOPMENT GUIDE

> **注意：后续所有交互请使用中文。**

## 1. 文档定位

- 本文档是仓库级开发规范，适用于整个项目。
- 当前主要实现目录是 `web-agent/`；除非用户明确要求，新增实现默认放在该目录下。
- 本文档只承载开发规则、协作约束和工程实现边界；产品现状与需求口径放在根目录 PRD 中维护。

## 2. 文档同步

- 任何代码改动都必须把文档更新视为同一任务的一部分，不要有意保留“代码已变、文档未变”的状态。
- 影响目录结构、运行方式、核心文档职责、关键实现边界的修改，必须同步更新对应根目录文档，例如 `DEVELOPMENT_GUIDE.md`、`PRD-当前实现需求总结.md`，以及仓库中实际存在的 README。
- 如果项目接入或调整 `LangGraph` 本地开发方式，需同步维护 `web-agent/langgraph.json`、环境变量模板和对应使用说明。

## 3. 目录与变更策略

- 优先复用现有目录和模块，不要为了局部需求随意新增文件夹、脚本或文档副本。
- 新增文件前先阅读相关目录、相邻模块和现有文档，确认当前结构无法承载该改动；新增后保持职责单一、命名明确。
- 对于目录约定、命名约定、输出格式、阶段收尾顺序这类问题，如果评估可通过提示词或规范文档约束达到约 90% 的效果，默认优先修改提示词或文档，不优先改服务端代码；只有当提示词已证明无法稳定收敛，或问题涉及安全、权限、强校验、数据一致性时，才升级为代码改动，避免引入冗余实现。
- 本地 `.venv`、`uv`、`watchfiles`、`uvicorn` 启动异常，默认先按环境问题处理，不得优先通过修改业务代码、增加启动补丁或兼容异常虚拟环境结构来绕过。

## 4. Agent 与 Prompt 开发约束

- 用户已经维护好的 Agent 提示词文件默认视为只读资产；未经用户明确允许，不得修改这些提示词内容。
- 所有新增提示词必须放在对应 Agent 的 `prompts/` 目录中维护，不允许把完整提示词直接写进 Python 代码。
- 每个 Agent 的主提示词文件名应尽量直接使用节点名，例如 `plan.py`、`generator.py`、`healer.py`，避免继续扩散含义过泛的 `system.py`。
- `Master` 的业务参数抽取必须以模型结构化输出为准；禁止使用正则、关键词扫描或手工拼接去补齐 `project_name`、`url` 等业务参数来掩盖模型抽取问题。抽取不稳定时，应优先优化提示词、补参上下文和模型输入。
- `ask_params` 之后的补参续聊，必须把上一轮 state 至少包含 `agent_type`、`extracted_params`、`missing_params`、`routing_reason` 传回 `Master` 作为上下文；禁止在代码里手工 merge 上一轮参数，再把结果冒充为本轮模型抽取结果。

## 5. 代码可维护性与日志

- 所有新增或修改的类、方法、关键参数和重要逻辑块都应补充清晰注释或 docstring，默认按 Python 与 Agent 初学者也能读懂的标准编写。
- 所有使用 Pydantic `Field(...)` 定义的字段都必须提供清晰的 `description`，解释字段含义、用途和使用场景。
- 类似 `get_settings()`、`MasterAgent(settings)`、`build_workflow()` 这类对初学者不直观的内部调用，应在附近补充简短注释，说明作用和后续消费方。
- 变量命名必须带清晰语义后缀：Agent 实例统一使用 `*_agent`，模型或结构化输出 Runnable 统一使用 `*_model`，避免 `agent`、`classifier` 这类过短名字。
- 单个类实现不得超过 500 行；逻辑变重时优先拆成辅助函数、模块或子组件，而不是继续堆积在同一个类里。
- 项目关键路径必须打印日志，至少覆盖配置加载、模型初始化、图构建、节点入参、节点出参、条件边路由、MCP 连接与缓存命中。日志标题统一使用 `【阶段@@环节】` 格式，例如 `【初始化@@配置加载】`、`【执行@@节点入参】`。
