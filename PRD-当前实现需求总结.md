# Web AutoTest Agent 当前实现需求总结 PRD

基于仓库现状整理日期：2026-04-28

## 1. 文档目的

本文档用于基于当前代码实现，梳理 `master`、`plan` 及相关模块的实际业务需求与产品边界。文档重点描述系统现在已经具备的业务能力、用户使用流程、输出结果和阶段边界，不展开底层开发实现细节。

## 2. 产品概述

Web AutoTest Agent 是一个面向 Web 自动化测试场景的多阶段智能体系统。当前版本的核心目标不是一次性完成“从需求到脚本到修复”的全流程，而是先稳定完成两件事：

1. 把用户请求准确识别并分流到正确阶段。
2. 在测试规划阶段，基于真实页面探索生成可落地的测试计划文档。

当前系统已经形成了“入口识别与补参 -> 计划生成 -> 后续阶段预留”的主链路，其中 `master` 和 `plan` 已具备实际业务能力，`generator` 和 `healer` 已具备接入位与边界定义，但尚未进入完整交付阶段。

## 3. 当前版本范围

### 3.1 已实现范围

- 识别用户请求属于测试规划、脚本生成、脚本修复，还是普通问答。
- 从用户表达中提取关键业务信息。
- 在信息不足时，按缺失项逐轮追问，而不是直接误执行。
- 进入测试规划阶段后，自动准备自动化项目目录。
- 基于真实页面探索生成 Markdown 测试计划。
- 按约定目录保存测试计划，并以“保存成功”作为完成标志。
- 为后续脚本生成与脚本修复预留独立阶段入口。

### 3.2 当前未完成范围

- 自动生成 Playwright 测试脚本的完整业务闭环尚未落地。
- 自动修复失败脚本的完整业务闭环尚未落地。
- 暂未形成“Plan 产出 -> Generator 批量落脚本 -> Healer 回归修复”的端到端自动化流水线。

## 4. 用户业务流程总览

### 4.1 总流程

1. 用户提出需求，例如“为某个页面生成测试计划”“根据用例写脚本”“修复失败用例”。
2. 系统先由 `master` 判断当前请求属于哪个阶段。
3. 如果信息不完整，系统不会直接执行，而是只追问当前最缺的一项关键信息。
4. 当请求属于 `plan` 且关键信息齐全时，系统进入测试规划阶段。
5. `plan` 阶段先准备工程目录，再进入页面探索，最后输出测试计划文档。
6. 当请求属于 `generator` 或 `healer` 时，当前版本会先正确识别并路由到对应阶段，但只返回阶段占位结果。
7. 当请求不属于自动化测试主链路时，系统按普通助手方式直接答复。

### 4.2 当前主路径

当前版本真正可执行的核心主路径为：

`用户提出测试规划需求 -> Master 识别与补参 -> Plan 真实探索页面 -> 保存测试计划`

## 5. 分模块需求

以下模块说明按实际业务推进顺序展开。

### 5.1 Master 模块

#### 5.1.1 模块定位

`master` 是整个系统的入口层，负责把自然语言请求变成可执行的业务判断。它的价值不是直接产出测试资产，而是先把任务识别清楚，减少后续阶段误执行。

#### 5.1.2 需要解决的问题

- 用户表述可能模糊，系统需要先判断当前到底是“做计划”“写脚本”“修脚本”还是“普通问答”。
- 用户一次可能只提供部分信息，系统需要判断哪些信息已经具备、哪些还缺失。
- 补参过程中，系统需要记住上一轮上下文，避免把补充信息误当成一个全新任务。

#### 5.1.3 当前业务能力

- 支持识别五类结果：`plan`、`generator`、`healer`、`general`、`unknown`。
- 能从用户输入中提取工程名、工程目录、URL、功能点、测试用例、失败脚本等信息。
- 能根据阶段要求判断必填信息是否完整。
- 当信息不完整时，一次只追问一个字段，降低用户理解成本。
- 在补参续聊中，会沿用上一轮已经确认的任务类型和已识别信息继续判断。
- 对于明显不属于自动化测试链路的请求，直接转为普通问答。

#### 5.1.4 当前阶段规则

- `plan` 阶段当前必填信息为：`project_name`、`url`。
- `project_dir` 为可选信息，不作为阻塞执行的必填项。
- `feature_points` 为可选上下文，用于帮助聚焦重点测试范围。
- `generator` 当前以测试用例作为关键输入。
- `healer` 当前以失败脚本或失败信息作为关键输入。

#### 5.1.5 用户可感知表现

- 当用户说“帮我做测试计划”，系统会优先判断为 `plan`。
- 如果只给了 URL 没给工程名，系统会继续追问工程名，而不会直接开始执行。
- 如果用户是在补充上一轮缺失信息，系统会继续补齐原任务，不会轻易重置任务方向。
- 如果用户只是闲聊或问概念问题，系统会直接回答，而不是误进入自动化流程。

### 5.2 Plan 模块

#### 5.2.1 模块定位

`plan` 是当前版本最完整的业务执行模块，负责把“一个待测页面或功能需求”沉淀为“结构化、可保存、可交接的测试计划文档”。

#### 5.2.2 启动条件

系统在以下条件满足时进入 `plan`：

- `master` 已判断当前任务属于测试规划。
- 已有有效的工程名。
- 已有有效的页面 URL。

如果工程目录未提供，系统可以自动推导并准备默认工程目录。

#### 5.2.3 当前业务目标

- 基于真实页面而不是想象来理解待测对象。
- 梳理关键用户路径、边界场景、异常场景和校验场景。
- 将结果保存为 Markdown 测试计划，供后续脚本生成阶段使用。

#### 5.2.4 当前执行流程

1. 确认工程信息和待测页面地址。
2. 准备自动化工程目录。
3. 初始化页面探索环境。
4. 打开目标页面并进行真实探索。
5. 如果用户给了功能点，优先围绕这些功能点组织测试范围。
6. 在页面探索基础上补全关键路径、边界条件和异常场景。
7. 按约定结构生成 Markdown 测试计划。
8. 将测试计划保存到项目目录中。
9. 以“计划保存成功”作为阶段完成标准。

#### 5.2.5 计划内容要求

当前实现要求测试计划具备以下特征：

- 以页面真实探索结果为基础，不根据 URL 或常识臆造功能。
- 优先覆盖关键用户旅程。
- 同时覆盖正常路径、边界条件、异常处理、输入校验和关键导航路径。
- 每个场景包含清晰步骤和对应预期结果。
- 场景之间尽量独立，默认从全新或空白状态开始。
- 输出为适合团队共享的 Markdown 文档，而不是脚本代码。

#### 5.2.6 当前产出要求

当前系统要求 `plan` 阶段只产出测试计划，不产出脚本文件。计划文件采用自动化项目内的统一目录与命名规范，例如：

- 计划优先保存到 `test_case/aaaplanning_{plan-name}/`
- 计划文档名使用 `aaa_{plan-name}.md`
- 计划中的场景会为未来脚本预留稳定命名方式

这意味着 `plan` 阶段的职责是“定义怎么测”，而不是“直接写测试代码”。

#### 5.2.7 完成判定

当前版本对 `plan` 的完成判定较严格：

- 不是看自然语言是否说“完成”。
- 而是看测试计划是否真的保存成功。

如果只是完成了页面分析，但没有实际保存计划，则本阶段视为未完成。

### 5.3 自动化工程与计划资产管理

#### 5.3.1 模块定位

该部分能力用于保证 `plan` 不是只在对话里给出建议，而是把结果沉淀进一个可继续演进的自动化工程空间。

#### 5.3.2 当前业务能力

- 当用户未指定工程目录时，系统会按“默认自动化根目录 + 工程名”的方式准备项目目录。
- 当目标目录不存在时，系统会自动复制一个内置 demo 工程作为基础模板。
- 当目标目录已存在时，系统会复用该目录继续工作。
- 如果项目目录内存在 `web_standard.md`，系统会把其中的业务规范一并纳入当前计划阶段。

#### 5.3.3 当前业务价值

- 让测试计划有稳定落点，而不是散落在对话历史中。
- 让不同页面、不同业务可以按工程维度持续沉淀。
- 为后续脚本生成与修复阶段预留统一资产结构。

### 5.4 Generator 与 Healer 衔接模块

#### 5.4.1 模块定位

`generator` 和 `healer` 代表自动化测试生命周期中的后续阶段：

- `generator` 负责根据测试计划或测试用例生成脚本。
- `healer` 负责围绕失败脚本进行排查和修复。

#### 5.4.2 当前已实现范围

- `master` 已能识别这两个阶段的任务并完成路由。
- 两个阶段都已具备独立提示边界和工具边界。
- 两个阶段都已保留未来真实执行的接入位。

#### 5.4.3 当前阶段状态

- 当前版本进入 `generator` 时，会返回已进入脚本生成阶段的提示。
- 当前版本进入 `healer` 时，会返回已进入脚本修复阶段的提示。
- 这两个阶段目前还没有完整的业务产物输出。

#### 5.4.4 对整体业务的意义

这两个模块虽然尚未完整交付，但已经把整个产品链路预留成三段式结构：

`计划 -> 生成 -> 修复`

因此当前版本不是一次性方案，而是已经具备后续持续扩展的业务框架。

### 5.5 运行保障模块

#### 5.5.1 模块定位

该部分能力用于保证系统在真实执行过程中更稳定、更可恢复，而不是一次失败就完全中断。

#### 5.5.2 当前业务能力

- 对页面探索和自动化操作提供统一的可调用能力边界。
- 自动为缺少依赖的自动化工程补齐基础运行依赖。
- 当某次页面操作失败时，系统倾向于先调整策略、重新观察页面，再继续推进。
- 当某个阶段发生未处理异常时，系统会中止当前阶段并返回可读错误信息，但保留已有上下文。
- 对补参轮次设置上限，避免用户与系统陷入无休止追问。

#### 5.5.3 用户价值

- 降低因为环境不完整或操作失败导致的直接中断。
- 提升计划生成阶段的稳定性与可恢复性。
- 让用户更容易理解系统失败在什么位置，而不是只能得到模糊报错。

## 6. 当前版本边界

### 6.1 明确边界

- 当前真正完成的核心交付物是测试计划文档，不是自动化脚本全集。
- 当前对自动化测试业务的理解入口已经稳定，但后续执行链路仍处于阶段化建设中。
- 当前产品最适合的使用方式，是先用它完成页面测试规划，再逐步衔接后续脚本生成能力。

### 6.2 非目标

- 当前版本不承诺直接完成完整回归测试闭环。
- 当前版本不承诺自动修复所有失败脚本。
- 当前版本不以“代码生成能力最强”为目标，而以“任务识别准确、计划产物可靠”为优先目标。

## 7. 当前验收口径

### 7.1 Master 验收口径

- 能区分计划、生成、修复、普通问答等主要任务类型。
- 能抽取关键业务信息。
- 能在缺参时继续追问，并保持上下文连续。
- 能把完整请求路由到正确阶段。

### 7.2 Plan 验收口径

- 在工程名和 URL 完整时可正常启动。
- 能在真实页面探索基础上形成测试计划。
- 能把测试计划保存到自动化工程目录。
- 只有保存成功才算阶段完成。

### 7.3 当前版本整体验收口径

- 用户已经可以把系统当作“测试规划入口”稳定使用。
- 系统已经具备从自然语言请求进入真实计划产出的主通路。
- 后续 `generator`、`healer` 可在现有框架下继续补齐，而不需要重做入口与阶段边界。
