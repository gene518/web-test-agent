# Web AutoTest Agent 当前实现需求总结 PRD

基于当前工作区状态整理日期：2026-05-04

## 1. 文档目的

本文档以当前工作区代码为事实来源，梳理 Web AutoTest Agent 已具备的产品能力、用户流程、阶段边界、前端交互表现和验收口径。本文档描述现状，不把未落地设想提前写成已实现能力。

## 2. 产品概述

Web AutoTest Agent 是一个面向 Web 自动化测试场景的多阶段智能体系统，当前围绕三类 Specialist 运转：

1. `plan`：基于真实页面探索生成 Markdown 测试计划。
2. `generator`：基于测试计划生成 Playwright `.spec.ts` 脚本。
3. `healer`：围绕失败脚本执行、调试、修复并验证。

`master` 作为统一入口子图，负责意图识别、参数补全、普通问答处理和阶段链路调度。当前版本已经支持用 `requested_pipeline + pipeline_cursor` 在一次请求内顺序推进多阶段执行，也支持在同一对话内承接上游阶段产物继续下一阶段。

## 3. 当前版本范围

### 3.1 已实现范围

- 识别用户请求属于测试规划、脚本生成、脚本修复、普通问答或定时任务配置。
- 从自然语言中抽取工程名、工程目录、URL、功能点、测试计划文件、测试用例和待修复脚本。
- 当信息不足时通过 LangGraph `interrupt` 暂停补参；恢复后固定原意图，沿用上一轮阶段链上下文继续抽取和合并参数。
- 支持一次请求内的多阶段顺序执行，例如 `plan -> generator`、`generator -> healer`、`plan -> generator -> healer`。
- 支持同一对话内继承最近一次 `plan`、`generator`、`healer` 产物，作为后续阶段的默认输入来源。
- 自动解析和准备自动化工程目录，支持默认根目录、显式工程目录和内置 demo 模板复制。
- `plan` 阶段接入 Playwright Test MCP，探索真实页面并通过 `planner_save_plan` 保存测试计划。
- `generator` 阶段消费 `test_plan_files` 文件或目录，解析计划文件，接入生成相关 MCP 工具，并通过 `generator_write_test` 写入脚本。
- `healer` 阶段消费 `test_scripts` 文件或目录，接入运行与调试工具，并允许在当前项目目录内修改测试代码。
- 三个 Specialist 阶段都会输出结构化最终总结；Plan、Generator、Healer 的最终用户总结已经统一采用“计数 + 路径 + 明细 + 下一阶段建议输入”的表达方式。
- 长对话默认支持 999 轮；超过后由 Master 模型生成 `conversation_summary`，后续模型输入结合摘要和最近消息。
- 运行时支持 MCP 会话复用、工具白名单、结构化工具错误、查询范围限制、项目规范加载和统一日志链路。

### 3.2 当前边界

- 当前版本支持按阶段链自动顺序推进，但不额外引入一条独立于现有主图之外的新编排系统。
- `plan` 的完成判定最严格，必须观察到 `planner_save_plan` 成功才算完成。
- `generator` 与 `healer` 已具备真实执行路径，并监听关键工具事件组织阶段总结。
- Generator 关于计划目录、脚本目录和测试计划 md 收尾方式，当前仍以提示词约束为主，不作为服务端强校验规则写死。
- Master 补参协议仍使用 `interrupt/resume`，调用方需要用 `Command(resume=用户补充内容)` 恢复。

## 4. 用户业务流程

### 4.1 总流程

1. 用户提出自动化测试相关请求，例如生成测试计划、根据计划写脚本、调试失败脚本，或一句话要求连续执行多个阶段。
2. 主图进入 `master_graph_node`，由 `Master` 子图识别意图、抽取参数，并生成本轮希望执行的 `requested_pipeline`。
3. 如果缺少关键参数，`complete_params_node` 通过 `interrupt` 暂停并追问当前最关键的一项；恢复后继续补齐同一意图的参数。
4. 参数齐全后，系统进入 `plan`、`generator` 或 `healer`；若本轮定义了多阶段链，则在阶段完成后回流 Master，推进 `pipeline_cursor` 并进入下一阶段。
5. 各 Specialist 负责准备项目目录、加载项目规范、获取允许工具并执行各自阶段任务。
6. 阶段执行完成后，系统累计阶段摘要；当前轮结束时由 `finalize_turn_node` 统一拼装最终回复。
7. 用户在后续轮次继续说“继续生成脚本”“调试脚本通过”等请求时，系统会优先复用同一对话内最近的上游产物作为默认输入。

### 4.2 阶段链路能力

当前版本已支持以下两类推进方式：

- 同轮串联：用户一句话明确要求多阶段执行时，系统按 `requested_pipeline` 顺序推进，例如 `plan -> generator` 或 `plan -> generator -> healer`。
- 跨轮承接：用户先完成 `plan`，下一轮再说“继续生成测试脚本”；或先完成 `generator`，再说“调试脚本通过”，系统会优先继承最近一次相关产物继续执行。

## 5. 分模块现状

### 5.1 Master 模块

#### 模块定位

`master` 是系统入口子图，负责把用户自然语言请求转成可执行的阶段判断、结构化参数和阶段链路。

#### 当前业务能力

- 支持识别 `plan`、`generator`、`healer`、`general`、`scheduler`、`unknown`。
- 能抽取并归一化关键参数，过滤空字符串和 `null`、`none`、`nil`、`undefined` 等空值标记。
- 能根据结构化输出与用户原话共同构造 `requested_pipeline`。
- 能基于阶段规则重新计算缺失参数，避免完全依赖模型自由返回的 `missing_params`。
- 对补参续聊，会把 `agent_type` 固定为原意图，只从用户补充内容中抽取参数并合并到当前上下文。
- 对普通问答在 `Master` 子图内直接走测试专家回答，不误触发自动化测试阶段。
- 通过 `display_messages` 维护 UI 专用时间线消息；这些消息可见于前端，但不作为后续模型上下文输入。

#### 阶段必填规则

- `plan`：必须具备 `project_name` 和 `url`。
- `generator`：必须具备 `test_plan_files`，并且至少有 `project_name` 或 `project_dir` 之一。
- `healer`：必须具备 `test_scripts`，并且至少有 `project_name` 或 `project_dir` 之一。

### 5.2 Plan 模块

#### 模块定位

`plan` 负责把页面探索结果和用户需求沉淀为可交接、可继续生成脚本的 Markdown 测试计划。

#### 当前执行流程

1. 解析并准备自动化工程目录。
2. 加载项目内的 `web_standard.md` 业务规范（如果存在）。
3. 获取 Plan 允许使用的 Playwright Test MCP 工具。
4. 调用 `planner_setup_page` 初始化页面。
5. 使用 `browser_navigate` 打开目标 URL 并探索真实页面。
6. 结合用户提供的 `feature_points` 和页面探索结果组织场景。
7. 调用 `planner_save_plan` 保存 Markdown 测试计划。
8. 保存成功后按提示词约束关闭浏览器并停止。

#### 当前产物与总结

- `plan` 的核心产物是已保存的测试计划 md。
- `plan` 还会记录“待生成脚本规划”，用于描述后续目标脚本路径，但这些路径不是已落地脚本。
- 最终总结会显式区分：
  - 已保存测试计划
  - 待生成脚本规划
  - 用例明细
  - 下一阶段建议输入

#### 完成判定

`plan` 不以自然语言“完成”为准，而是以 `planner_save_plan` 工具成功事件为准。没有保存计划文件则视为阶段失败，并返回可读错误信息。

### 5.3 Generator 模块

#### 模块定位

`generator` 负责根据已确认的测试计划生成 Playwright 测试脚本，把计划资产转换为 `.spec.ts` 代码资产。

#### 当前执行流程

1. 解析自动化工程目录；目录不存在时按内置 demo 模板准备工程。
2. 将 `test_plan_files` 解析为项目目录内的绝对路径。
3. 如果输入是目录，优先展开 `aaa_*.md`，否则展开普通 Markdown 文件。
4. 拒绝项目目录外的计划路径，避免跨目录读取。
5. 加载项目规范、文件查询约束和 Generator 阶段提示词。
6. 获取 Generator 允许使用的 MCP 工具，包括页面操作、验证、`generator_setup_page`、`generator_read_log` 和 `generator_write_test`。
7. 使用事件流执行 Deep Agent，并监听 `generator_write_test` 的成功或错误状态。
8. 生成完成后将阶段原始结果交给总结层；如果写入后关闭浏览器触发预期关闭异常，则视为成功收尾。

#### 当前产物与目录规则

- 测试脚本通过 `generator_write_test` 写回当前自动化工程目录。
- Generator 阶段会记录来源测试计划、已生成脚本、脚本明细和下一阶段建议输入。
- 计划目录与脚本目录的收尾约定目前仍主要由 Generator 提示词约束，例如生成脚本后复制计划 md、清理旧计划目录，不把这部分行为当作服务端硬约束。

### 5.4 Healer 模块

#### 模块定位

`healer` 负责围绕已有失败脚本进行运行、调试、修复和验证，目标是让失败脚本恢复可用或按规则收敛为明确的 `test.fixme()`。

#### 当前执行流程

1. 解析自动化工程目录；目录不存在时按内置 demo 模板准备工程。
2. 将 `test_scripts` 解析为项目目录内的绝对路径。
3. 如果输入是目录，递归展开其中的 `.spec.ts` 文件。
4. 拒绝项目目录外的脚本路径，避免跨目录读写。
5. 加载项目规范、文件查询约束和 Healer 阶段提示词。
6. 获取 Healer 允许使用的 MCP 工具，包括 `test_run`、`test_debug`、`test_list` 和页面诊断工具。
7. 为 Deep Agent 绑定当前项目目录的读写权限，允许修复脚本写回项目。
8. 使用事件流执行调试与修复，并将阶段原始结果交给总结层。

#### 修复边界与总结

- 只允许读写当前 `project_dir` 下的文件。
- 优先针对 `test_scripts` 指定的脚本运行与调试，不默认扩大到整个工程。
- 修改后应重新运行相关脚本验证。
- 最终总结会显式给出调试目标脚本、实际变更文件、验证运行目标、脚本明细和下一阶段建议输入。

### 5.5 自动化工程与运行保障

#### 工程目录管理

- 默认自动化根目录由 `DEFAULT_AUTOMATION_PROJECT_ROOT` 控制。
- 未显式提供 `project_dir` 时，按 `默认根目录 / project_name` 解析工程目录。
- 显式提供相对 `project_dir` 时，相对默认自动化根目录解析。
- 目标目录不存在时，从内置 `an-autotest-demo` 模板复制基础工程。
- 目标目录已存在时复用原目录，不覆盖已有内容。

#### 运行保障

- Playwright Test MCP 工具按阶段白名单精确暴露。
- MCP 会话按 server 和 workspace 复用，减少重复连接。
- 工具错误会被包装为模型可见的结构化结果，便于模型调整策略。
- Specialist 内置文件工具被限制在当前项目目录内，并屏蔽 `node_modules`、`test-results` 和大体积调试产物。
- 日志覆盖配置加载、图构建、模型初始化、节点入参、节点出参、路由、MCP 连接、工具事件和关键完成事件。

## 6. 前端交互现状

### 6.1 线程与时间线展示

- 前端以 `display_messages` 作为用户可见时间线的基础状态，并将其与持久化消息、实时消息做去重合并。
- 人类消息、AI 消息、工具消息和补参消息会尽量保持在统一时间线中展示，避免重复渲染和中断丢失。
- 自定义 `display_messages` 事件会在前端按批次合并，减少频繁刷新造成的抖动。

### 6.2 人工中断与恢复

- Generic interrupt 在输入框顶部标题展示 `人工中断：{question}`。
- `question` 不再混在中间字段表格中重复展示。
- 用户提交补参文本后，前端会把该文本同时写入 `messages` 和 `display_messages` 的乐观时间线。

### 6.3 工具结果展示

- 工具调用参数和工具结果支持结构化展示。
- 大体积文本和结构化对象优先显示预览，再按需展开。
- 工具结果不是只展示原始大块字符串，而是支持更紧凑的可读摘要。

## 7. 当前版本验收口径

### 7.1 Master 验收口径

- 能区分计划、生成、修复、普通问答、定时任务和未知请求。
- 能抽取目标阶段所需参数，并在缺参时通过 `interrupt/resume` 固定原意图继续补参。
- 能基于用户请求生成单阶段或多阶段 `requested_pipeline`。
- 能在同轮多阶段执行时正确推进 `pipeline_cursor`，并在阶段链结束后统一汇总。

### 7.2 Plan 验收口径

- 工程名和 URL 完整时可启动。
- 能准备或复用自动化工程目录。
- 能基于真实页面探索生成测试计划。
- 只有 `planner_save_plan` 成功才算阶段完成。
- 阶段总结必须区分已保存测试计划和待生成脚本规划。

### 7.3 Generator 验收口径

- 能接受计划文件或计划目录作为输入。
- 能将输入解析为当前项目目录内的 Markdown 计划文件。
- 能加载 Generator 阶段上下文、项目规范和允许工具。
- 能通过 Deep Agent 执行脚本生成流程，并监听 `generator_write_test` 状态。
- 阶段总结必须包含来源测试计划、已生成脚本、脚本明细和下一阶段建议输入。

### 7.4 Healer 验收口径

- 能接受脚本文件或脚本目录作为输入。
- 能将输入解析为当前项目目录内的 `.spec.ts` 文件。
- 能加载 Healer 阶段上下文、项目规范和允许工具。
- 能在当前项目目录内读写文件，支持脚本修复和复测。
- 阶段总结必须包含调试目标脚本、变更文件、验证运行目标、脚本明细和下一阶段建议输入。

### 7.5 整体验收口径

- 用户可以按阶段使用系统完成测试计划、脚本生成和失败脚本修复。
- 用户也可以在一次请求中明确要求多个阶段连续执行，系统会按阶段链顺序推进。
- 同一对话内可承接上游阶段产物继续执行下游阶段。
- General、Plan、Generator、Healer 的用户可见结果都经过最终总结。
- 前端能稳定展示中断恢复、结构化工具结果和基于 `display_messages` 的可见时间线。

## 8. 非目标

- 本文档不承诺自动修复所有产品缺陷或所有不稳定测试。
- 本文档不把提示词约束误写成服务端强校验能力。
- 本文档不声明新增外部 API、额外工具白名单或独立的新编排引擎。
