# Web AutoTest Agent 当前实现需求总结 PRD

基于仓库现状整理日期：2026-04-30

## 1. 文档目的

本文档基于当前代码实现，梳理 Web AutoTest Agent 已具备的产品能力、用户流程、阶段边界和验收口径。文档以代码现状为事实来源，重点描述业务行为和产物边界，不展开底层实现细节。

## 2. 产品概述

Web AutoTest Agent 是一个面向 Web 自动化测试场景的多阶段智能体系统。当前版本已经形成三类 Specialist 分工：

1. `plan`：基于真实页面探索生成 Markdown 测试计划。
2. `generator`：基于测试计划生成 Playwright `.spec.ts` 脚本。
3. `healer`：围绕失败脚本运行、调试、修复并验证。

`master` 作为统一入口子图，负责识别用户意图、补全参数、处理通用测试问答和阶段路由。当前图结构仍是单次请求路由到一个目标阶段，不额外声明自动串联三阶段的新编排节点。

## 3. 当前版本范围

### 3.1 已实现范围

- 识别用户请求属于测试规划、脚本生成、脚本修复、普通问答或未知请求。
- 从自然语言中抽取工程名、工程目录、URL、功能点、测试计划文件、测试用例和待修复脚本。
- 信息不足时通过 LangGraph `interrupt` 暂停当前节点，请用户补充缺失参数；恢复后固定原意图继续抽取和合并参数。
- `general` 请求在 Master 子图内由通用测试专家提示词回答，并经过最终总结包装后返回。
- 自动解析和准备自动化工程目录，支持默认根目录、显式工程目录和内置 demo 模板复制。
- `plan` 阶段接入 Playwright Test MCP，探索真实页面并通过 `planner_save_plan` 保存测试计划。
- `generator` 阶段消费 `test_plan_files` 文件或目录，解析计划文件，接入生成相关 MCP 工具，并通过 `generator_write_test` 写入脚本。
- `healer` 阶段消费 `test_scripts` 文件或目录，接入运行与调试工具，并允许在当前项目目录内修改测试代码。
- `plan`、`generator`、`healer` 阶段完成后只把最终总结作为用户可见消息返回，MCP 过程消息保留为内部阶段结果。
- 长对话默认支持 999 轮；超过后由 Master 模型生成 `conversation_summary`，后续模型输入结合摘要和最近消息。
- 运行时支持 MCP 会话复用、工具白名单、结构化工具错误、查询范围限制、项目规范加载和统一日志链路。

### 3.2 当前边界

- 当前图结构按一次请求进入一个阶段；没有新增自动执行 `Plan -> Generator -> Healer` 全流程的独立编排节点。
- `plan` 的完成判定最严格，必须观察到 `planner_save_plan` 成功才算完成。
- `generator` 已具备真实生成执行路径，并会监听 `generator_write_test` 工具事件；正常完成结果会进入最终总结。
- `healer` 已具备真实调试与修复执行路径；修复结果会进入最终总结。
- Master 补参协议使用 `interrupt/resume`，调用方需要用 `Command(resume=用户补充内容)` 恢复。

## 4. 用户业务流程总览

### 4.1 总流程

1. 用户提出自动化测试相关请求，例如生成计划、根据计划写脚本、修复失败脚本。
2. 外层主图进入 `master_graph_node`，由 Master 子图判断请求类型并抽取后续阶段需要的参数。
3. 如果缺少关键参数，`complete_params_node` 通过 `interrupt` 暂停并追问当前最关键的一项；恢复后继续补齐同一意图的参数。
4. 参数齐全后，系统根据意图进入 `plan`、`generator` 或 `healer`。
5. 目标 Specialist 准备项目目录、加载项目规范、获取允许的 MCP 工具并执行对应阶段任务。
6. 阶段执行完成后，系统调用 Master 模型生成最终总结，只把总结作为用户可见消息返回。
7. Specialist 回流到 Master 后只执行收尾守卫并结束当前轮，避免把阶段过程消息重新分类。

### 4.2 三阶段链路

当前产品能力可以支持按用户请求分阶段推进：

`Plan 生成测试计划 -> Generator 生成脚本 -> Healer 修复失败脚本`

这表示三个阶段都已有对应运行能力；不表示当前图会在一次请求中自动无条件串联三阶段。

## 5. 分模块需求

### 5.1 Master 模块

#### 模块定位

`master` 是系统入口子图，负责把用户自然语言请求转成可执行的阶段判断和结构化参数。子图内部按节点拆分为意图判断、参数补全和通用测试问答。

#### 当前业务能力

- 支持识别 `plan`、`generator`、`healer`、`general`、`unknown`。
- 能抽取并归一化关键参数，过滤空字符串和 `null`、`none`、`nil`、`undefined` 等空值标记。
- 能基于阶段规则重新计算缺失参数，避免完全依赖模型自由返回的 `missing_params`。
- 对补参续聊，会把当前 `agent_type` 固定为原意图，只从用户补充内容中抽取参数并合并到当前上下文。
- 对普通问答在 Master 子图内直接走测试专家回答，不误触发自动化测试阶段。
- 对 Specialist 回流设置 `return_to_master` 收尾守卫，避免同一轮重新分类。

#### 阶段必填规则

- `plan`：必须具备 `project_name` 和 `url`。
- `generator`：必须具备 `test_plan_files`，并且至少有 `project_name` 或 `project_dir` 之一。
- `healer`：必须具备 `test_scripts`，并且至少有 `project_name` 或 `project_dir` 之一。

### 5.2 Plan 模块

#### 模块定位

`plan` 负责把一个页面或功能需求沉淀为可交接、可继续生成脚本的 Markdown 测试计划。

#### 启动条件

- `master` 已识别为测试规划任务。
- 已有有效的 `project_name`。
- 已有有效的被测页面 `url`。
- `project_dir` 可选；未提供时按默认自动化根目录和工程名推导。

#### 当前执行流程

1. 解析并准备自动化工程目录。
2. 加载项目内的 `web_standard.md` 业务规范，如果该文件存在。
3. 获取 Plan 允许使用的 Playwright Test MCP 工具。
4. 调用 `planner_setup_page` 初始化页面。
5. 使用 `browser_navigate` 打开目标 URL 并探索真实页面。
6. 结合用户提供的 `feature_points` 和页面探索结果组织场景。
7. 调用 `planner_save_plan` 保存 Markdown 测试计划。
8. 保存成功后按提示词约束关闭浏览器并停止。

#### 完成判定

`plan` 不以自然语言“完成”为准，而是以 `planner_save_plan` 工具成功事件为准。没有保存计划文件则视为阶段失败，并返回可读错误信息。

### 5.3 Generator 模块

#### 模块定位

`generator` 负责根据已确认的测试计划生成 Playwright 测试脚本，把计划资产转换为 `.spec.ts` 代码资产。

#### 启动条件

- `master` 已识别为脚本生成任务。
- 已有 `test_plan_files`，可以是一个或多个 Markdown 文件，也可以是包含计划文件的目录。
- 已有 `project_dir`，或已有 `project_name` 以便按 Plan 相同规则推导项目目录。

#### 当前执行流程

1. 解析自动化工程目录；目录不存在时按内置 demo 模板准备工程。
2. 将 `test_plan_files` 解析为项目目录内的绝对路径。
3. 如果输入是目录，优先展开 `aaa_*.md`，否则展开普通 Markdown 文件。
4. 拒绝项目目录外的计划路径，避免跨目录读取。
5. 加载项目规范、文件查询约束和 Generator 阶段提示词。
6. 获取 Generator 允许使用的 MCP 工具，包括页面操作、验证、`generator_setup_page`、`generator_read_log` 和 `generator_write_test`。
7. 使用事件流执行 Deep Agent，并监听 `generator_write_test` 的成功或错误状态。
8. 生成完成后将 Deep Agent 原始结果交给最终总结；如果写入后关闭浏览器触发预期关闭异常，则视为成功收尾。

#### 产物规则

- 测试脚本通过 `generator_write_test` 写回当前自动化工程目录。
- 计划目录和脚本目录遵循现有 Generator 提示词中的约定。
- 运行时只允许读取当前项目目录，写脚本由 MCP 生成工具承担。

### 5.4 Healer 模块

#### 模块定位

`healer` 负责围绕已有失败脚本进行运行、调试、修复和验证，目标是让失败脚本恢复可用或按规则收敛为明确的 `test.fixme()`。

#### 启动条件

- `master` 已识别为脚本修复任务。
- 已有 `test_scripts`，可以是一个或多个 `.spec.ts` 文件，也可以是包含脚本的目录。
- 已有 `project_dir`，或已有 `project_name` 以便按 Generator 相同规则推导项目目录。

#### 当前执行流程

1. 解析自动化工程目录；目录不存在时按内置 demo 模板准备工程。
2. 将 `test_scripts` 解析为项目目录内的绝对路径。
3. 如果输入是目录，递归展开其中的 `.spec.ts` 文件。
4. 拒绝项目目录外的脚本路径，避免跨目录读写。
5. 加载项目规范、文件查询约束和 Healer 阶段提示词。
6. 获取 Healer 允许使用的 MCP 工具，包括 `test_run`、`test_debug`、`test_list` 和页面诊断工具。
7. 为 Deep Agent 绑定当前项目目录的读写权限，允许修复脚本写回项目。
8. 使用事件流执行调试与修复，并将原始结果交给最终总结。

#### 修复边界

- 只允许读写当前 `project_dir` 下的文件。
- 优先针对 `test_scripts` 指定的脚本运行与调试，不默认扩大到整个工程。
- 修改后应重新运行相关脚本验证。
- 如果确认属于产品缺陷，可按提示词规则使用 `test.fixme()` 收敛。

### 5.5 自动化工程与运行保障

#### 工程目录管理

- 默认自动化根目录由 `DEFAULT_AUTOMATION_PROJECT_ROOT` 控制。
- 未显式提供 `project_dir` 时，按 `默认根目录 / project_name` 解析工程目录。
- 显式提供相对 `project_dir` 时，相对默认自动化根目录解析。
- 目标目录不存在时，从内置 `an-autotest-demo` 模板复制基础工程。
- 目标目录已存在时复用原目录，不覆盖已有内容。

#### 运行保障

- Playwright Test MCP 工具按阶段白名单精确暴露。
- MCP 会话按 server 和 workspace 复用，减少重复连接。
- 工具错误会被包装为模型可见的结构化结果，便于模型调整策略。
- Specialist 内置文件工具被限制在当前项目目录内，并屏蔽 `node_modules`、`test-results` 和大体积调试产物。
- 日志覆盖配置加载、图构建、模型初始化、节点入参、节点出参、路由、MCP 连接、工具事件和关键完成事件。
- 阶段最终总结统一覆盖“用户要求什么、分析怎么做、如何做、完成了什么”。

## 6. 当前版本验收口径

### 6.1 Master 验收口径

- 能区分计划、生成、修复、普通问答和未知请求。
- 能抽取目标阶段所需参数。
- 能在缺参时产生 `interrupt`，并在 `Command(resume=...)` 后固定原意图继续补参。
- 能把完整请求路由到正确 Specialist。
- 能在 Specialist 回流时结束当前轮，不重新分类阶段结果。

### 6.2 Plan 验收口径

- 工程名和 URL 完整时可启动。
- 能准备或复用自动化工程目录。
- 能基于真实页面探索生成测试计划。
- 只有 `planner_save_plan` 成功才算阶段完成。

### 6.3 Generator 验收口径

- 能接受计划文件或计划目录作为输入。
- 能将输入解析为当前项目目录内的 Markdown 计划文件。
- 能加载 Generator 阶段上下文、项目规范和允许工具。
- 能通过 Deep Agent 执行脚本生成流程，并监听 `generator_write_test` 状态。
- 能把生成阶段结果包装为最终总结返回给用户。

### 6.4 Healer 验收口径

- 能接受脚本文件或脚本目录作为输入。
- 能将输入解析为当前项目目录内的 `.spec.ts` 文件。
- 能加载 Healer 阶段上下文、项目规范和允许工具。
- 能在当前项目目录内读写文件，支持脚本修复。
- 能通过运行与调试工具推进失败脚本修复，并把结果包装为最终总结返回给用户。

### 6.5 整体验收口径

- 用户可以按阶段使用系统完成测试计划、脚本生成和失败脚本修复。
- 三个 Specialist 都有明确输入、工具边界、项目目录边界和结果返回方式。
- General、Plan、Generator、Healer 的用户可见结果都经过最终总结。
- 当前文档与代码实现保持一致，明确 Generator 和 Healer 都已有对应运行能力。

## 7. 非目标

- 本文档不声明新增一条自动无条件执行三阶段的 LangGraph 编排链路。
- 本文档不承诺自动修复所有产品缺陷或所有不稳定测试。
- 本文档不声明新增工具白名单或新增外部 API。
